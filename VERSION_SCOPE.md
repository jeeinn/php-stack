# PHP-Stack v0.1.0 版本定位说明

## 📌 核心定位

**PHP-Stack v0.1.0** 是一个**环境配置管理与迁移工具**，专注于帮助 PHP 开发者：
1. **可视化配置环境** - 替代手动编辑 `.env` 和 `docker-compose.yml`
2. **统一管理镜像源** - 加速国内开发体验
3. **备份与恢复环境** - 快速迁移开发环境到新机器

---

## ✅ v0.1.0 包含的功能

### 1. 可视化环境配置
- GUI 界面选择服务类型（PHP、MySQL、Redis、Nginx）及版本
- 端口配置与实时冲突检测
- PHP 扩展多选配置
- 自动生成 `.env` 文件和 `docker-compose.yml`
- 保留用户自定义变量
- 支持多 PHP 版本独立服务
- 创建 dnmp 风格目录结构（services/、data/、logs/）

**用户价值**: 
- ⏱️ 节省手动配置时间
- 🔒 避免配置错误
- 📝 配置文件完全可见可编辑

### 2. 统一镜像源管理
- 5 个预设方案（阿里云、清华、腾讯云、中科大、官方默认）
- 4 类镜像源独立配置（Docker Registry、APT、Composer、NPM）
- 连接测试功能（3 秒超时）
- 一键应用预设或单独配置

**用户价值**:
- 🚀 加速 Docker 镜像拉取
- 🌏 优化国内开发体验
- 🔧 灵活配置各类别源

### 3. 环境备份
- ZIP 格式备份包，包含 `manifest.json`
- SHA256 文件完整性校验
- 可选：数据库导出、项目文件、vhost 配置、日志
- Tauri 事件进度通知
- 部分失败容错处理

**用户价值**:
- 💾 完整备份开发环境
- 🔐 确保备份完整性
- 📦 便于团队协作和迁移

### 4. 环境恢复
- 备份预览（manifest 解析、文件统计）
- SHA256 完整性验证
- 端口冲突检测与自动分配
- 配置文件还原、数据库 SQL 执行
- 进度通知与错误汇总

**用户价值**:
- 🔄 快速恢复开发环境
- 🆕 新机器快速上手
- ⚠️ 智能处理端口冲突

## ❌ v0.1.0 不包含的功能

### 1. 软件管理中心（多版本一键安装）

**为什么不包含**:
- **复杂性高**: 需要维护版本清单、处理依赖关系、管理容器生命周期
- **偏离核心**: v0.1.0 聚焦于“配置管理”而非“软件安装”
- **用户可控**: 开发者通常已有偏好的镜像来源和管理方式

**替代方案**:
- 用户自行从 Docker Hub 拉取所需镜像
- 使用 `docker pull php:8.2-fpm` 等命令
- 在 EnvConfigPage 中配置已存在的镜像版本

**未来可能**:
- 如果用户需求强烈，可在 V3.0 考虑
- 需要重新评估产品定位和复杂度

### 2. 虚拟主机管理（Nginx 站点配置）

**为什么不包含**:
- **专业性高**: Nginx 配置涉及复杂的 rewrite、SSL、负载均衡等
- **场景多样**: 不同项目需求差异大，难以通用化
- **透明性原则**: v0.1.0 强调配置文件可见可编辑

**替代方案**:
- 用户手动编辑 `services/nginx/conf.d/` 下的 `.conf` 文件
- 使用备份功能保存 vhost 配置
- 参考 dnmp 项目的 Nginx 配置示例

**未来可能**:
- 可提供常用配置的模板库
- 但不做完整的 GUI 管理界面

## 🎯 设计理念

### 1. 轻量级
- **专注核心**: 只做配置管理和环境迁移
- **避免复杂**: 不做容器编排、软件安装等重型功能
- **快速启动**: 安装包小，启动快，资源占用低

### 2. 透明性
- **配置可见**: 生成的 `.env` 和 `docker-compose.yml` 完全可见
- **可编辑**: 用户可随时手动修改配置文件
- **无黑盒**: 不隐藏任何实现细节

### 3. 兼容性
- **dnmp 兼容**: 目录结构和配置格式与 dnmp 保持一致
- **标准优先**: 使用标准的 Docker Compose 语法
- **团队协作**: 配置文件可直接提交到 Git

### 4. 可靠性
- **测试覆盖**: 72 个单元测试全部通过
- **属性测试**: 12 个设计属性通过 proptest 验证
- **容错设计**: 备份/恢复支持部分失败，不中断整体流程

## 🔄 用户使用流程

### 典型场景 1: 新项目搭建

```
1. 打开 EnvConfigPage
2. 选择 PHP 8.2、MySQL 5.7、Nginx
3. 配置端口和扩展
4. 点击"应用配置"
5. 手动拉取 Docker 镜像: docker pull php:8.2-fpm
6. 使用 docker-compose up -d 启动
```

### 典型场景 2: 环境迁移

```
1. 在旧机器上打开 BackupPage
2. 选择备份选项（配置、数据库、项目文件）
3. 点击"创建备份"，保存 ZIP 文件
4. 在新机器上打开 RestorePage
5. 选择备份 ZIP 文件
6. 点击"开始恢复"
7. 手动拉取所需 Docker 镜像
8. 使用 docker-compose up -d 启动
```

### 典型场景 3: 镜像源加速

```
1. 打开 MirrorPanel
2. 选择"阿里云全套"预设
3. 点击"测试连接"验证可用性
4. 点击"应用"保存配置
5. 后续 Docker 操作自动使用阿里云镜像源
```

## 💡 常见问题

### Q1: 为什么不提供一键安装 PHP/MySQL？

**A**: v0.1.0 的定位是“配置管理工具”而非“软件安装工具”。我们假设用户：
- 已经了解 Docker 基本概念
- 知道如何拉取 Docker 镜像
- 希望保持对环境的完全控制

这样可以保持工具的轻量和透明，避免引入过多复杂性。

### Q2: 如何管理多个 PHP 版本？

**A**: 
1. 在 EnvConfigPage 中添加多个 PHP 服务（如 PHP 7.4 和 PHP 8.2）
2. 配置不同的端口
3. 应用配置后，手动拉取对应版本的镜像
4. 使用 `docker-compose up -d` 启动所有服务

### Q3: 如何配置 Nginx 虚拟主机？

**A**:
1. 手动编辑 `services/nginx/conf.d/your-site.conf`
2. 参考 dnmp 项目的配置示例
3. 使用备份功能保存配置
4. 重启 Nginx 容器使配置生效: `docker-compose restart nginx`

### Q4: 备份的数据库能完整恢复吗？

**A**: 
- 当前版本中，mysqldump 为占位符实现
- 用户可以手动导出数据库: `docker exec mysql-container mysqldump -u root -p --all-databases > backup.sql`
- 恢复时手动导入: `docker exec -i mysql-container mysql -u root -p < backup.sql`
- 未来版本会完善自动化的 mysqldump 执行

### Q5: v0.1.0 适合哪些用户？

**A**: 
- ✅ 熟悉 Docker 基本概念的 PHP 开发者
- ✅ 需要在多台机器间迁移开发环境的团队
- ✅ 希望可视化管理 Docker 配置的用户
- ✅ 追求轻量级工具，不喜欢重型 IDE 插件的用户

- ❌ 完全不了解 Docker 的新手
- ❌ 需要一键安装所有软件的懒人模式用户
- ❌ 需要复杂容器编排的企业级用户
